Rules & conventions

Where the rules live and the constraints to respect when building new pages (e.g. the new self-service homepage).

  • Tokens source of truth: the :root block in styles.css. Never introduce a new hex — add or reuse a token. The README brand-token table is outdated.
  • CSS lives in three files by zone: styles.css (homepage + shared: buttons, cards, nav, hero, timeline, install agent), results.css (case studies — metric cards etc.), blog.css (blog posts).
  • Class naming: the site uses plain BEM-ish names — .btn, .card, .agent__icon, .card--link. The design-system handoff ships --es-* tokens and .es-btn/.es-card; those must be remapped to the names above when applied here (see issue #27 to end this drift).
  • Nav & footer exist in three places — index.html, _layouts/post.html, resources/index.html. Change all three together.
  • Case studies are not Jekyll — results/* and this page are hand-built static HTML (no Liquid, no front matter).
  • Fonts use root-relative @font-face paths (/fonts/…) so they load from sub-pages like /styleguide/ and /results/slug/.
  • Motion honors prefers-reduced-motion — every hover lift, wiggle, and the install sweep are disabled under it. Match that for new motion.
  • Mobile integrity is gated by the Huginn Playwright suite (no horizontal overflow at iPhone SE width). Keep new components within the viewport.
  • Open follow-ups: #25 (Stripe checkout + setup-page states), #26 (self-service homepage repositioning), #27 (source tokens/components from the shared design system).
